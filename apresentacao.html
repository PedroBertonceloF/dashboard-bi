Avaliação de IAs para Geração de Código-Fonte

WEB-02 · Dashboard de BI/Analytics a partir de CSV · Tópicos Especiais em Computação

Claude (Anthropic) Gemini CLI (Google)

Problema

Contexto

Este trabalho faz parte da disciplina de Tópicos Especiais em Computação e tem como objetivo avaliar, de forma experimental e comparativa, ferramentas de IA para geração de código-fonte. O grupo resolveu o mesmo problema utilizando duas ferramentas distintas, registrando o processo, os prompts e os resultados obtidos.

Problema escolhido — WEB-02

Dashboard de BI/Analytics a partir de CSV (KPIs + filtros)

Construir uma aplicação web que permita carregar dados de um arquivo CSV e gerar indicadores-chave de desempenho (KPIs) e visualizações interativas com filtros.

Funcionalidades mínimas exigidas

  • Upload de CSV com pré-visualização das primeiras linhas
  • Seleção e validação de colunas (data, categoria, valor)
  • Limpeza básica: tratar vazios, datas inválidas, separador decimal, espaços em branco
  • Dois gráficos: série temporal e barras por categoria
  • Filtros: intervalo de datas e categoria (multi-seleção)
  • Exportação: relatório CSV com KPIs

Critérios de aceitação

  • KPIs conferem com cálculo manual em uma amostra de validação
  • Filtros atualizam KPIs e gráficos corretamente
  • Erros de CSV malformado ou coluna ausente são tratados com mensagem clara
  • Não inventar números: resultados devem derivar exclusivamente do CSV
As ferramentas avaliadas foram Claude (Anthropic) e Gemini CLI (Google), utilizando o mesmo conjunto de requisitos e laboratórios de avaliação para ambas.

Metodologia

Abordagem geral

Cada ferramenta recebeu os mesmos requisitos e foi utilizada para construir uma solução completa e independente para o problema WEB-02. O processo foi documentado via diário de prompts, com registro das iterações, falhas e correções.

Ferramentas avaliadas

  • Claude (Anthropic) — modelo claude-sonnet-4-5, operado como assistente de chat com intervenções manuais de código
  • Gemini CLI (Google) — agente autônomo com acesso ao terminal, capaz de executar comandos, criar arquivos e corrigir erros em loop

Laboratórios de avaliação

Defini cinco laboratórios para medir aspectos específicos de cada solução. Minha intenção não era apenas verificar se cada IA conseguia "fazer funcionar", mas sim isolar dimensões de engenharia que costumam separar um protótipo de algo pronto para produção — preocupações que eu, como desenvolvedor, normalmente teria que lembrar de pedir explicitamente, e quis ver se a IA já antecipava isso sozinha:

  1. Lab 1 – Performance de Ingestão (format='mixed', cache=True para 1M+ linhas): quis saber se a IA pensa em escala sem eu pedir. Qualquer solução resolve um CSV de 100 linhas; o que me interessava era se ela já antecipava 1M+ linhas, usando parsing vetorizado em vez de iterar linha a linha.
  2. Lab 2 – Edge Cases (CSV malformado com on_bad_lines='skip', colunas extras preservadas): quis ver como cada IA lida com dados do mundo real, que quase sempre vêm sujos. O ponto era saber se uma linha malformada seria tratada como exceção isolada (descartando só ela) ou se derrubaria o processamento do arquivo inteiro.
  3. Lab 3 – Segurança (JWT com expiração, bcrypt com salt, proteção contra SQL Injection via ORM, segredos em variáveis de ambiente): quis avaliar se a IA aplica boas práticas de segurança por padrão, mesmo sem eu pedir item por item. Para mim, esse é o lab que mais separa "código que funciona" de "código que eu deixaria ir para produção".
  4. Lab 4 – UX e Conectividade (proxy Vite para resolver CORS, feedback visual de erros no frontend com React + Recharts): quis ver como cada IA resolve um problema de integração real dentro de uma restrição que eu impus ("sem mexer no Python"), e se entregava uma experiência de usuário coerente, com feedback de erro e gráficos.
  5. Lab 5 – Resiliência (robustez do pipeline de limpeza sem perda de dados não mapeados): complementa o Lab 2 — aqui eu queria saber se a limpeza de dados é robusta e não destrutiva, preservando o que eu não mapeei e se sustentando em variações de formato sem precisar reescrever tudo.

Em resumo, cada laboratório simula uma exigência que eu faria depois que o "MVP" já estivesse pronto — e meu objetivo era medir o quanto cada ferramenta já antecipava essas exigências sem que eu precisasse pedir explicitamente.

Histórico de Prompts

Principais prompts utilizados durante o desenvolvimento com cada ferramenta, em ordem cronológica.

Claude
Prompt 1

"Eu tenho uma API FastAPI que recebe um CSV. Eu preciso que você escreva o arquivo dataset_ingestion.py. Ele deve ler o CSV e limpar os dados de forma estrita: se a Data, Categoria ou Valor estiverem vazios ou errados, delete a linha inteira. Importante: Se o CSV tiver colunas extras que eu não mapeei, elas DEVEM ser mantidas no arquivo final. Remova também espaços em branco de todos os textos. Como você faria para que isso fosse extremamente rápido mesmo em arquivos com 1 milhão de linhas?"

Prompt 2

"Com base no meu modelo de usuário que tem email e hashed_password, implemente um módulo de segurança completo em Python. Ele deve usar JWT para tokens e bcrypt para as senhas. Os tokens devem expirar em 30 minutos. Além disso, realize uma auditoria: existe algum risco de SQL Injection no meu código se eu usar SQLAlchemy?"

Prompt 3

"Minha API Python roda na porta 8001 e meu React na 5173. Eu estou tentando logar e dá erro de 'Connection Refused' ou 'CORS'. Como eu resolvo isso sem mexer no Python? Além disso, crie o componente de Dashboard em React que faça o upload do CSV e mostre os KPIs e os Gráficos (use Recharts)."

Gemini CLI
Prompt 1 — Entrevista (grill-with-docs)

"leia o arquivo COMOFAZER que esta nessa PASTA, e use o grill-with-docs"

O agente leu os requisitos e usou a skill grill-with-docs para entrevistar o usuário, fazendo perguntas sobre o escopo. Identificou um conflito entre o escopo do WEB-02 e o checklist de segurança da disciplina e sugeriu proativamente criar login/registro. As respostas dadas pelo usuário nessa entrevista alimentaram os próximos passos.
Prompt 2 — Geração do PRD

"/to-prd"

Com base nas respostas da entrevista, o agente gerou o PRD do projeto, com critérios de aceitação mensuráveis e as decisões de arquitetura (ADRs).
Prompt 3 — Quebra em issues

"/to-issues"

O agente quebrou o PRD em issues numeradas, cada uma com título, descrição, critérios de aceitação e dependências — é a Issue #2 dessa quebra que aparece no próximo prompt.
Prompt 4 — Implementação (TDD)

"vamos fazer a 2 com tdd"

Referente à Issue #2 (upload de CSV). O agente criou a tabela datasets, a rota de upload e o teste automatizado, seguindo o ciclo RED → GREEN do TDD.
Prompt 5 — Debug de performance

"acho que escolhi um arquivo grande, esta demorando mt para processar"

O agente identificou que o Pandas usava inferência padrão de datas (lenta), corrigiu para format='mixed' e vetorizou as operações de string, deixando o processamento instantâneo.
Prompt 6 — Debug de ambiente (Windows)

"ele nao faz login nem faz registro também..."

O agente diagnosticou dois problemas de infraestrutura no Windows: o "QuickEdit Mode" do terminal pausando o Uvicorn (ensinou a destravar com ESC) e um processo fantasma preso na porta 8000 (mudou autonomamente para a porta 8001).
Prompt 7 — Melhoria (higienização)

"quero fazer com que eu limpe não só os dados que eu escolhi, mas todos os dados limpos da tabela inteira"

O agente usou a skill interview-me para entender a real intenção e implementou via TDD: preserva todas as colunas originais, remove espaços em branco de todos os textos e aplica exclusão rigorosa só nas colunas mapeadas.
Prompt 8 — Ajuste visual

"não quero que a legenda do grafico fique para fora..."

Ajustou paddings de CSS e margens internas dos gráficos Recharts para que eixos e legendas ficassem legíveis em qualquer resolução.
Prompt 9 — Documentação final

"crie uma pasta chamada execução e coloque so o que escrever para exectuar o back e o front"

Criou a pasta execução/ e o arquivo COMO_RODAR.md, centralizando o manual de instruções do projeto.

Conclusão

O que funcionou bem

O Claude demonstrou uma postura "enterprise" desde a primeira resposta no Lab 3: gerou SECRET_KEY via variável de ambiente, bcrypt com rounds explícitos, headers de segurança extras (CSP, HSTS, X-Frame-Options) e uma suíte de 48 testes automatizados (31 deles focados só em segurança). No Lab 1, já entregou uma solução pronta para grandes volumes de dados sem precisar de iterações adicionais.

O Gemini CLI mostrou autonomia real nos prompts G-01/G-02/G-03: gerou o PRD, os ADRs e o detalhamento em issues e gravou os arquivos sozinho no projeto, sem precisar que o resultado fosse copiado e colado manualmente. No Lab 2, aderiu exatamente ao gabarito esperado, usando on_bad_lines='skip' para descartar apenas linhas malformadas.

O que falhou ou exigiu correção

O Gemini CLI deixou a SECRET_KEY fixa (hardcoded) no código-fonte do security.py — algo que viola diretamente o checklist de segurança do plano de projeto ("segredos fora do repositório"). Além disso, o Lab 1 só atingiu um resultado satisfatório depois de mais rodadas de conversa para ajustar o tratamento de datas e o volume de linhas suportado.

O Claude, no Lab 2, não implementou on_bad_lines: uma única linha de CSV com número de colunas inconsistente faz o upload inteiro ser rejeitado com erro 400, em vez de descartar apenas a linha problemática como o gabarito sugeria.

Grau de interferência necessária

A interferência necessária teve naturezas diferentes em cada ferramenta. Com o Gemini CLI, a interferência foi mais em nível de prompting — foi preciso conversar mais para refinar o Lab 1 até chegar a um resultado aceitável. Com o Claude, a interferência foi mais em nível de integração manual — o código gerado já vinha mais completo, mas precisava ser copiado e colado nos arquivos do projeto.

Qualidade e organização do código gerado

Há uma diferença marcante de "verbosidade defensiva" entre os dois. O security.py gerado pelo Claude tem docstrings extensas, seções comentadas, validação de força de senha e funções utilitárias extras (refresh de token, cálculo de tempo restante de expiração). Já o equivalente do Gemini CLI é mais enxuto e direto ao ponto, porém funcional. A diferença na cobertura de testes automatizados (48 vs. 17) reforça essa impressão: o Claude tende a produzir mais artefatos de verificação junto com o código de produção.

Riscos identificados

  • Gemini CLI: SECRET_KEY hardcoded no código-fonte — risco real de vazamento de segredo se o repositório for público ou compartilhado.
  • Claude: rejeição total do arquivo no Lab 2 pode ser um problema em datasets grandes com poucas linhas problemáticas, obrigando o usuário a limpar o CSV manualmente antes do upload.
  • Ambos: nenhuma das duas implementações configurou CORSMiddleware no FastAPI — dependem inteiramente do proxy de desenvolvimento do Vite, o que não funcionaria se o frontend fosse servido separadamente em produção. Fica como ponto de trabalho futuro para ambos.

Síntese

Não houve um "vencedor" absoluto — cada ferramenta se destacou em frentes diferentes do mesmo problema. O Claude entregou, desde a primeira resposta, um módulo de segurança mais robusto e completo (segredos via variável de ambiente, validação de senha, headers extras, 48 testes automatizados) e lidou melhor com grandes volumes de dados no Lab 1. Já o Gemini CLI se destacou pela autonomia — gerou e gravou sozinho PRD, ADRs e issues no projeto (G-01 a G-03) — e aderiu mais fielmente ao gabarito de edge cases do Lab 2 (on_bad_lines='skip').

Em outras palavras: o Claude tendeu a produzir código mais "defensivo" e pronto para produção logo de cara, enquanto o Gemini CLI tendeu a economizar trabalho de planejamento e organização do projeto, executando tarefas que normalmente exigiriam intervenção manual do desenvolvedor.

Recomendações de uso

  • Usar agentes autônomos como o Gemini CLI para tarefas de planejamento e scaffolding (PRDs, ADRs, quebra em issues, criação de estrutura de arquivos), onde a autonomia de gravar arquivos diretamente economiza tempo de trabalho manual.
  • Usar o Claude em módulos sensíveis a segurança e regras de negócio críticas, onde a tendência a gerar código mais documentado, validado e testado reduz o retrabalho.
  • Independentemente da ferramenta, revisar manualmente configurações de segurança (segredos, chaves, variáveis de ambiente) — o caso da SECRET_KEY hardcoded no Gemini CLI mostra que esse tipo de falha pode passar despercebido mesmo em código funcional.
  • Conferir os critérios de aceitação do edital linha a linha contra o código gerado — diferenças sutis (como o tratamento de on_bad_lines no Lab 2) só aparecem nessa checagem detalhada.

Limitações deste estudo

A comparação foi feita sobre um único problema (WEB-02 — Dashboard de BI a partir de CSV) e com versões específicas de cada ferramenta em um momento pontual; resultados podem variar com prompts diferentes, novas versões dos modelos ou problemas de outra natureza. Além disso, parte da análise (Labs 2, 3 e 5) foi feita por inspeção do código gerado, sem executar testes automatizados de ponta a ponta com arquivos CSV malformados reais — o que seria um próximo passo natural para validar as conclusões na prática.

No fim, as duas ferramentas se mostraram capazes de produzir uma aplicação funcional e razoavelmente completa a partir da mesma especificação, mas com personalidades distintas: uma mais cautelosa e "documentação em primeiro lugar", outra mais autônoma e orientada a execução. Para uso profissional, o ideal parece ser combinar as duas abordagens — aproveitar a autonomia de um agente para tarefas repetitivas e a robustez do outro para código crítico — sempre com revisão humana antes de ir para produção.
